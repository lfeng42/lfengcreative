Mom's Almost Home!





Background

At Georgia Tech, I took a course called "Media Device and Architecture" in which we learned how to program in low-level C by creating GameBoy games and get a taste of what assembly level programming was like. Through our assignments, we learned how to connect hardware and software worked together by using C code to manipulate device hardware. We were also introduced to basic game programming methods and game design techniques. We learned how to use the program Usenti to turn sprite images into bitmaps that could then be used in GameBoy games by indexing the pixels in the images.



Task

For our final project, we were tasked with creating a GameBoy game in C using what we had learned in class. There were no limitiations as to what kind of gameplay was necessary or how the game looked, but it had to have the following components in the game:

  • At least 2 animated sprites
  • 1 tiled background larger than 512x512 pixels
  • Win/Lose state screens
  • Splash screen with menu and instructions
  • Ability to pause the game
  • 2 sounds that play at the same time, 1 sound that loops
  • Cheat to make the game faster to finish (not just skipping to Win state



Process

I brainstormed many ideas, but at the end I decided to go with a basic game that centered around a boy who had to clean up his house before his mom arrived home.

Gameplay Overview and Brainstorm


Description:

You move around with arrow keys to pick up trash around the house. If you accidentally collide with a vase, you have to return it to the nearest shelf before you can collect any other trash so you don't accidentally break it. If you run into a dog, the timer decreases by 5 seconds. There's a timer counting down so you have to make sure you collect all trash before mom comes home!


Win state:

Collect all the trash before the timer runs out (60 seconds) and mom comes home.


Lose state:

You don't collect all the trash before the time runs out (60 seconds) and mom comes home.


Collisions:

If a player collides into a piece of trash, it "picks" it up. Colliding into a vase forces you to collide with a shelf to place it on the shelf. Character turns red if it's currently holding a vase.


Controls:
  • Up: Player moves in the up direction
  • Down: Player moves in the down direction
  • Left: Player moves in the left direction
  • Right: Player moves in the right direction
  • Start: Pause game, see pause menu that allows you to restart game or return to main menu.


Audio:
  • Opening Music: Chugging, bouncy music
  • Collecting Trash: Shuffling sound
  • Bumping into Dog: Barking sound
  • Background: Happy bumbling music

Implementation

I drew all the graphics, screens, and sprites using Usenti's paint feature and I created a sprite sheet bitmap. In addition, to make collisions for the furniture and walls on the map to prevent the player from being able to walk through them, I created a collision map in Usenti as well. State machines were also important to the basis of my game code because it made it easy to transition from screen to screen. It was more organized in the code and most efficient to implement.



Final Product

For the look of the game, I went with a simple 8-bit appearance to mimic the old school, retro appearance that many games of the GameBoy time had.

Reflection

Taking this class gave me my first exposure to how hardware and software connected together. In addition, although it was not the intention of the course, game design and programming were what I felt I gained the most from the course itself. Working on this GameBoy project gave me a glimpse into the effort it takes into making games, and I truly respect the effort game makers put into their products.


Lessons Learned


  • Making user experiences for games is much different from other mediums.
    • User experiences on digital platforms like websites or mobile apps usually only involve the visuals. However, with games, there are other components that go into making a good game experience. For example, there's the abstract component of whether it is "fun" or not. Is it challenging enough that people will be invested to play it? Is the gameplay interesting enough to the users? There are also other factors that go into the user experience like the intuitiveness of the many controls in the GameBoy and the atmosphere that the sound (or the lack of it) makes in the experience.


If I could do this all again...


If I could do this again, I would make the graphics much more detailed and give them dimension. I was unfamiliar with drawing in 8-bit, pixel art because it was my first time working with Usenti. The text when drawn in with Usenti was difficult to read because it was so pixelated and blocky, and the art did not have the detail that I wanted. I was more focused on programming the backend of the project that I dedicated less time to the graphics. Next time, I would like to play around with using Photoshop to create more detailed graphics and import them into Usenti to create a more aesthetically pleasing experience.
